What happened this session
---------------------------------------------------------------------

1. Manual testing of the S3 domain surfaced real UX friction and
   feature gaps, each scoped via AskUserQuestion before writing code:

   - `ui.PickList` gained type-ahead filtering as a *generic* feature
     (used at ~30 call sites, not just buckets): non-numeric,
     non-empty, non-'n'/'p' input narrows the current list by a
     case-insensitive label substring match; blank clears the filter.
   - `inventory.ListBuckets` now returns buckets sorted alphabetically
     by name -- also fixed a latent bug where the enrichment
     goroutine fan-out made bucket order nondeterministic run to run.
   - Manage Bucket Lifecycle Policies gained local validation that a
     transition's days must be strictly less than its rule's
     expiration days (and vice versa) -- closes the gap opened by
     2026-07-08's real-AWS verification (see DECISIONS.md, "0.0.1
     scope" and the earlier "Clear a bucket's lifecycle configuration
     via DeleteBucketLifecycle..." entries).
   - Manage Bucket Lifecycle Policies gained a read-only "View rule
     details" action (Add/Edit/Remove/View/Back) -- loops back to the
     action menu after viewing, not back to the caller (a regression
     caught and fixed mid-session).
   - Two new destructive S3 workflows: Delete Bucket (refuses a
     non-empty bucket, reports the object count, `ConfirmDestructive`
     typed-bucket-name gate) and Delete Objects by Prefix (bulk delete
     under a key prefix or the whole bucket, same confirmation
     pattern). Both required adding `DeleteBucket` to
     `awsclient.S3API` + its logging wrapper + the shared fake.
   - CloudFront removed entirely from the domain picker
     (`DomainActions.CloudFront`, the menu entry, its
     `NotYetImplemented` wiring in `main.go`) at the user's explicit
     request -- postponed to a much later version, not deleted from
     DESIGN.md/PLAN.md's design.

   Full file-by-file detail in this session's hand-off,
   agents/hand-off/2026-07-09T180000Z-awstools-0.0.1-release-prep.spmd.

2. Evaluated `charmbracelet/bubbletea`, `charmbracelet/huh`, and
   `peak/s5cmd/v2` for a future UI/UX pass -- by `go get`-ing all three
   into a scratch module and reading actual source, not just docs or a
   marketing article the user linked. Conclusions (full detail in the
   hand-off and DECISIONS.md's "0.0.1 scope" entry):

   - `bubbletea`: Elm-architecture, async message loop -- mismatched
     with termlib's and every workflow function's blocking, linear
     style. Would be a full rewrite of ~40 wizard functions plus their
     tests, not a library swap.
   - `huh` (built on bubbletea): closer fit. `huh.Run(field)` is
     blocking/synchronous like `ui.Prompt`/`PickList`/`Confirm`; its
     `Select` field has built-in incremental filtering; every field's
     `RunAccessible(w, r)` is structurally identical to this project's
     `newPipeEditor` test pattern. Real cost: pulls in
     bubbletea+bubbles+lipgloss and ~18 transitive modules, vs.
     termlib's ~2,800 dependency-free lines.
   - `s5cmd/v2`: not a good import (CLI-framework coupling, aws-sdk-go
     v1 vs. this project's v2) but its `storage.S3.MultiDelete` batch-
     `DeleteObjects`-in-chunks-of-1000 pattern is worth reimplementing
     natively later (aws-sdk-go-v2 already exposes `DeleteObjects`) --
     logged to TODO.md's "Nice to have," not started.
