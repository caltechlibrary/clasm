2. Did the real-AWS manual verification pass, per the plan's
   Verification checklist, against the real Caltech Library AWS account
   (074441911104, us-west-2) -- confirmed with the user first given the
   account holds real production infrastructure (theses, archives,
   media, caltechdata, etc.), using a `rsdoiel-awsops-verify-*` naming
   prefix for throwaway resources. Drove the actual compiled binary

   if this technique is needed again). Verified: Create Bucket (all

   upload, then a locally-deleted-file pass exercising the separate
   delete confirm), Browse/Manage Objects (with and without a prefix
   filter, metadata, delete), a guided backup lifecycle policy
   (round-tripped via GetBucketLifecycleConfiguration), and the generic
   editor's add/edit/remove on the internal bucket. Everything checked
   independently via `aws s3api`/`aws s3` CLI calls, not just the tool's
   own success messages.

3. Found and fixed one real bug via this verification (exactly what the
   real-AWS pass is for): removing a bucket's *last* lifecycle rule
   failed, because `PutBucketLifecycleConfiguration` requires a non-empty
   `Rules` field (an AWS SDK client-side validation, before any network
   call) and there was no code path to actually clear a lifecycle
   configuration. Fixed per this project's test-before-fix practice: a
   failing test reproduced it first, then added `DeleteBucketLifecycle`
   to `S3API`/`logging_s3.go`, and `ManageBucketLifecyclePolicies` now
   calls that instead of `PutBucketLifecycleConfiguration` when the
   modified rule set is empty. Re-verified against real AWS after the
   fix -- `GetBucketLifecycleConfiguration` correctly reports
   `NoSuchLifecycleConfiguration` after removing the last rule.
   Recorded in DECISIONS.md, "Clear a bucket's lifecycle configuration
   via DeleteBucketLifecycle, not an empty PutBucketLifecycleConfiguration".

4. Also found, but deliberately did NOT fix (worked around instead), a
   second real constraint: AWS rejects a guided backup policy whose
   expiration-days is less than its transition-days (an object must
   transition to cheaper storage before it can expire). The guided flow
   currently has no local validation for this -- it surfaces AWS's raw
   error message. Left as an open, documented gap (TODO.md,
   DECISIONS.md) rather than fixed in this pass, since it's a smaller,
   narrower issue than the last-rule-removal bug and wasn't blocking
   the rest of verification (worked around by choosing valid test
   values: transition < expiration).

5. All throwaway AWS resources (3 buckets, a handful of test objects)
   were deleted and independently confirmed gone
   (`aws s3api list-buckets` for the `rsdoiel-awsops-verify-*` prefix
   returns empty). No production bucket was read-modified or touched.

6. Docs updated: PLAN.md's Phase 20 section now has a "Real-AWS
   verification" paragraph; DECISIONS.md has a new entry for the
   DeleteBucketLifecycle fix; TODO.md's Phase 20 line now says
   "real-AWS verified" and gained a new open item for the transition/

   is invisible to normal usage).
