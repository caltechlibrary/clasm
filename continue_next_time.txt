2. User began live-testing bin/awsops against real AWS mid-Phase-15 and
   reported two real gaps, both fixed:
   - Key pair/security group/subnet name entry was free-text-only;
     typing a security group *name* instead of an ID caused a real
     `InvalidParameterCombination` AWS error. Fixed: security groups
     and subnet are now pick lists fetched from the AMI's region
     (ec2:DescribeSecurityGroups/DescribeSubnets); key pair name was
     tried as a pick list too but reverted to free text after real
     testing showed a flat list of every key pair in the account
     (16+ entries, mostly irrelevant) was noise, not help.
   - A validator (`requireNonEmpty`) written for one prompt (cloud-init
     YAML) with a hardcoded field-specific message got reused generically
     across 8 unrelated prompts without generalizing the message --
     fixed to a field-agnostic message.
   - Name tag prompt moved to right after the AMI pick (was at the end
     of the flow) per feedback that naming the instance should come
     early.

3. Added -debug: a JSONL log of every AWS SDK call (internal/debuglog's
   nil-safe *DebugLog + internal/awsclient's WrapEC2/WrapSSM/WrapS3/
   WrapSTS logging decorators, one shared generic helper). Modeled on
   ~/Laboratory/harvey's own --debug feature, but decorates the existing
   narrow client interfaces instead of hand-instrumenting call sites, so
   every AWS action is covered automatically. IMPORTANT: CreateKeyPair's
   response carries private key material and is redacted to a fixed
   marker before logging -- do not remove that special case if touching
   logging_ec2.go.

4. Added inline "create a new key pair" support: typing "new" at the Key
   pair name prompt (instead of the user asking "what is Key pair
   name?" then "can we support creating one?") switches into a sub-flow
   that calls ec2:CreateKeyPair (ED25519, PEM), saves the private key to
   ~/.ssh/<name>.pem at 0600, and uses that name for the launch. A name
   collision re-prompts; any other error propagates normally.

5. Used the new -debug log to diagnose a real launch failure live: AMI
   ami-0da49db6a772dda02 isn't ENA-enabled, and t3.micro requires ENA --
   confirmed from the JSONL record's exact RunInstances params + AWS
   error, not guessed.

6. User asked about pre-flight sanity-checking + a retry-on-error loop
