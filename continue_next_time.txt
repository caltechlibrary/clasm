4. THE BULK OF THE SESSION: drove Backup Archive & Trim to a fully
   working real-AWS run against `newauthors` (i-0c4c81336aea33d27,
   ~87GB in /opt/rdm_sql_backups) targeting a new bucket,
   sql-backups.library.caltech.edu. Found and fixed THREE sequential
   real bugs, each diagnosed from the -debug JSONL log first (never
   guessed):

   BUG 1 -- MovedPermanently. awsops' single global S3 client was
   hardcoded to `cfg.Regions[0]` (us-west-1); the bucket lives
   elsewhere, and HeadBucket/HeadObject 301 with no useful detail on a
   region mismatch. FIX: `BucketRegion()` calls `s3:GetBucketLocation`
   (works from a client scoped to ANY region) to discover the bucket's
   real region, then a `newS3Client(region)` factory builds a client
   actually scoped there for every subsequent S3 call. Also added a new
   `CheckS3BucketAccess` preflight (`s3:HeadBucket`) right after the "S3
   bucket" prompt, aborting fast if the bucket is unreachable at all.

   BUG 2 -- AccessDenied. The instance's attached role,
   `AWS-QuickSetup-SSM-DefaultEC2MgmtRole-us-west-2` (AWS's generic
   auto-attached SSM management role), had zero S3 permissions. This
   was an AWS-console fix, not code: walked the user through IAM ->
   Roles -> that role -> inline policy granting `s3:PutObject`/
   `s3:ListBucket` on the whole bucket. Deliberately bucket-wide (not
   scoped to one instance's prefix), since the user explicitly wants
   this to work for "any of our EC2 instances that run SQL" -- flagged
   clearly that this means every instance sharing that role can write
   anywhere in the bucket, accepted as a reasonable tradeoff for a
   small trusted fleet.

   BUG 3 -- silent false-negative on large files. With CLI + IAM both
   fixed, small files worked but a ~972MB file still reported FAIL
   despite a real, successful `aws s3 cp` in the debug log. Root cause:
   `aws s3 cp`'s own \r-updated progress meter filled
   `ssm:GetCommandInvocation`'s documented 24,000-character stdout cap,
   pushing this project's own `printf 'OK\t...'` signal off the end
   before it was ever captured -- a genuine success silently
   misreported as failure. FIX: `--only-show-errors` added to the
   remote `aws s3 cp` call (suppresses progress noise, keeps real error
   text on stderr). This broke two test fakes that matched on the old
   exact command text -- fixed (caught immediately by Go's 10-minute
   test timeout, not a production hang).

   Also, while on this thread: namespaced every upload by instance
   (`s3://bucket/<instance-name>/<filename>`, falls back to instance ID
   if untagged) since the bucket serves multiple systems and files
   could otherwise collide -- and switched uploads to one
   `ssm:SendCommand` per file (was one batched script) so a live
   "N/M (bytes of total) - OK/FAIL key" progress line prints per file.

5. Ran the fixed workflow against a SECOND instance, `data-new`, and hit
   the exact same "AWS CLI not found" failure again (confirmed from the
   debug log, not guessed). The user's own framing: "That should be an
   explicit error we check for much like with shell scripts we check if
   a command exists before trying to execute it." Added
   `CheckAWSCLIAvailable` (`internal/workflow/backup_cli_check.go`):
   runs `command -v aws` via SSM immediately after picking the
   instance, before ANY other prompt, aborting fast with a clear,
   actionable error naming the instance if it's missing -- instead of
   every subsequent upload silently reporting FAIL. Confirmed IAM
   policy from Bug 2 already covered `data-new` (same shared role) --
   installing the CLI was the only missing piece there, and the full
   run then succeeded.
