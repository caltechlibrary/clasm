What happened this session
---------------------------------------------------------------------

1. Implemented and shipped Phase 19 (Key Management domain): List,
   Create, Import, Delete Key Pair. TDD throughout, mirroring existing
   Compute-domain patterns (internal/inventory/keypairs.go mirrors
   instances.go/images.go; keypair_delete.go mirrors remove_ami.go's
   dry-run/dependency-check/type-to-confirm shape; keypair_create.go
   reuses Phase 15.2's existing createNewKeyPairInteractive rather than
   duplicating it). One real design decision: Key Management's own
   refresh independently calls inventory.ListInstances (not just
   ListKeyPairs) so Delete Key Pair's dependency-check warning is
   correct even if Key Management is entered before Compute in a given
   run -- see DECISIONS.md. User manually confirmed all four operations
   working against real AWS.

2. Found and fixed a real bug -- properly, not just worked around. The
   user's real-AWS test of Import Key Pair produced garbled, repeated
   prompt text with typed input invisible. Root cause: an ~180-char
   prompt label (embedding a usage hint) broke termlib's
   LineEditor.Prompt raw-mode redraw, which assumes the whole prompt
   fits on one terminal row and repaints via a bare "\r" (can't move up
   to a previous row). Per the user's explicit instruction ("if there's
   a bug in termlib, fix that"), root-caused and fixed at
   ~/Laboratory/termlib/lineeditor.go with a new splitSafePrompt helper
   (8 new unit tests there). This also retroactively fixes two
   PRE-EXISTING awstools prompts with the same latent risk
   (create_key_pair.go, create_instance_profile.go) that had never been
   reported broken. The user committed+pushed the termlib fix (commit
   354195d) and repointed awstools' go.mod to a real pseudo-version pin
   (v0.0.10-0.20260708184214-354195d36c57) instead of my temporary
   local-path replace directive, which I then removed. The awstools-side
   prompt itself was simplified back to a short label per further user
   feedback, with the longer guidance moved into the validation error
   message instead.

3. Established practice, saved to memory: write a test that reproduces
   a reported bug and CONFIRM IT FAILS against the current code, before
   writing the fix -- not test-alongside-or-after. Where the bug lives
   in raw-mode TTY behavior unreachable by the pipe-based test harness
   (both awstools' and termlib's own test harnesses always hit the
   non-raw fallback() path for non-TTY input), say so explicitly and
   test the closest provable invariant instead, then ask for manual
   interactive re-verification.

4. Designed Phase 20 (S3 domain) end to end, without writing any code,
   per this project's design-then-implement discipline:
   - Researched existing patterns first (an Explore agent + direct
     reads of internal/awsclient/s3.go, the BucketRegion/newS3Client
     region-resolution pattern from Backup Archive & Trim, PickList's
     pagination).
   - Asked and got answers to four scope questions up front: defer the
     Feature 19 public-read bucket-policy opt-out (CloudFront doesn't
     exist yet to hand off to); add an optional key-prefix filter to
     Feature 21 (Browse/Manage Objects) since real buckets like
     sql-backups.library.caltech.edu hold many objects across many
     per-instance prefixes; a Purpose-tag-driven guided/generic UX
     split; full multi-rule CRUD.
   - Mid-planning, the user described a real gap: three bucket "types"
     the tool should make easy -- website (already designed), backup
     (needs expiration + transition-to-cheaper-storage policies,
     matching what Backup Archive & Trim already produces), and
     internal (unpredictable policy needs). This became new Feature
     21.1, "Manage Bucket Lifecycle Policies" -- asked four more
     clarifying questions (all confirmed): tag-and-remember bucket
     purpose (not just a creation-time wizard); optional-prefix policy
     scope; one Purpose-branching feature, not two menu entries;
     multiple named rules with full CRUD, not a single policy.
   - Wrote all of this into DESIGN.md (Feature 18 expanded with a
     purpose wizard; Feature 17 reads the tag back; new Feature 21.1,
     numbered as a decimal insertion specifically to avoid renumbering
     CloudFront's Features 22-26 and ~15 existing cross-references,
     mirroring PLAN.md's own Phase 15.1-15.26 convention; new Security
     Consideration #13) and two new DECISIONS.md entries recording
     every judgment call. PLAN.md's Phase 20 rewritten to match; effort
     raised ~10h -> ~16h.
   - User reviewed and confirmed the design/decisions/plan. Session
     ended there -- next session starts directly on implementation.
