What happened this session
---------------------------

1. User ran ./ec2_ami_manager.bash for real against production AWS and
   hit three real, previously-undetected bugs:
   - show_pick_list()'s eval had unbalanced quoting + a misplaced [@]
     subscript, crashing the interactive picker with "unexpected EOF".
     Fixed: eval "items=(\"\${${arr_name}[@]}\")"
   - The AMI-name validation regex broke under BSD grep + a UTF-8 locale
     ("invalid character range"). Fixed by pinning LC_ALL=C and moving
     the hyphen to an unambiguous bracket position.
   - create_ami_from_instance()'s --tag-specifications string builder
     produced malformed AWS CLI shorthand, silently failing create-image
     with NO AMI created anywhere (confirmed via read-only
     describe-images across all 4 regions) — this was the actual root
     cause of "I don't see the AMI actually get generated."
   Also implemented the Phase 7 main menu (never actually wired up
   despite PLAN.md claiming it was done), a default AMI-name suggestion,
   and a wider AMI-name display column.

2. After fixing bug #3, the user asked whether Go/Python/Deno would be
   faster than continuing in Bash. Decision: retarget to Go. This led to
   a long collaborative design session that grew the v1 scope far beyond
   the original 4-operation Bash tool.

3. Real findings that shaped the new design (not hypothetical):
   - newauthors (i-0c4c81336aea33d27) has 87GB in /opt/rdm_sql_backups,
     ~980MB/day, zero rotation — the measured cause of the
     "over-provisioned disk / slow cloning" problem.
   - Tagging today is inconsistent (a "project" tag on some instances,
     no dedicated Environment tag anywhere — prod/test lives only in the
     instance NAME string).
   - newauthors/authorstest both have real, recoverable cloud-init
     user-data (proven via ec2:DescribeInstanceAttribute).
   - cloud-init-examples' invenio-rdm.yaml has already drifted from what's
     actually deployed.

4. DESIGN.md/DECISIONS.md/PLAN.md fully rewritten for Go. CLI binary
   named "awsops" (repo stays "awstools"). Final v1 primitive list (12
   features, in final menu order — see DESIGN.md Core Features):
   1) Unified Resource Listing, 2) Create Instance from AMI (now with
   file-path user-data + cloud-init completion check), 3) Create Instance
   from Cloud-Init YAML, 4) Start Instance, 5) Stop Instance,
   6) Terminate Instance (flags DeleteOnTermination volumes), 7) Manage
   Tags, 8) Create AMI from Instance, 9) Remove AMI, 10) Show/Export
   Cloud-Init, 11) Backup Archive & Trim (two-phase upload-then-
   independently-verify, never trusts the instance's self-report),
   12) Project/Environment Tagging Convention (cross-cutting policy, not
   an action — distinct from #7).

5. Confirmed against real AWS behavior: an AMI's Name is permanently
   immutable via the API (no rename operation exists or can exist short
   of CopyImage + deregister). Only Description is mutable post-creation.

6. Every confirmation-gated workflow (PLAN.md Phases 4-13) is required to
   separate "build a resolved params struct" from "execute against AWS,"
   with the confirmation/dry-run gate as its own reusable function — not
   because record/replay ships in v1, but so a deferred "Recorded
   Scripts" YAML-playbook feature can be added later without reopening
   finished code.
