management menu and display of buckets with the huh module."

Scope (confirmed via research this session, see DECISIONS.md/hand-off
for detail):

   picker: Show resource lists / Create Bucket / Configure Static

   Policies / Delete Bucket / Back to domain picker) -> `huh.Select`.
2. The bucket-selection step inside `ConfigureBucketWebsite`
   (bucket_website.go), `ManageBucketLifecyclePolicies`

Also folds in a wording/navigation fix the user asked for directly
(TODO.md, "UX improvements" section, added 2026-07-09): "exit",
"cancel", and "back to ..." are used inconsistently across this app's
menus. The user is OK standardizing on "quit" to close a menu (which
should just back you up one level in the hierarchy), with "cancel"
reserved for aborting an in-progress action, not for closing a menu.
Concretely for this piece of work: right now, aborting (Ctrl-C/Esc)
*anywhere* inside the S3 menu exits the whole program (`isExitSignal`
treats it as a real quit) -- change this so aborting the newly-huh-
converted top-level S3 menu returns `ErrBackToDomainPicker` (back one
level, to the domain picker) instead. Scope this behavior change to
just the S3 menu for now; leave a note that Compute/Key Management's
still-termlib menus should get the same fix when they're migrated
later -- don't silently change their behavior as a side effect of this
work.

Open question to resolve FIRST, before writing more untested huh code:
`internal/workflow/object_browser.go` (the only existing huh usage in
this codebase) currently has zero test coverage. huh's real
interactive `Field.Run()`/`Form.Run()` isn't pipe-testable the way
termlib's `newPipeEditor` pattern is (see confirm_test.go). Investigated
this session but not resolved: huh's accessible-mode text format is
confirmed from source --
- `Select.RunAccessible`: writes the title, then `"%d. %s\n"` per
  option (1-indexed), then `"Enter a number between 1 and N: "`; reads
  one line.
- `Confirm.RunAccessible`: writes title + `" [y/N]"`/`" [Y/n]"`; reads
  one line, parses y/yes/n/no/blank.
- Both go through `bufio.Scanner` reading one newline-delimited line
  per field, and (unlike termlib) never propagate an error on EOF --
  a pipe test harness must supply complete valid lines, not rely on an
  error signal.
- `Form.runAccessible` iterates every group/field in sequence, ignoring
  each field's own error return, writing a blank line after each.

Not yet tried: whether `huh.NewForm(...).WithAccessible(true).
WithInput(pipeReader).WithOutput(&buf).Run()` (forcing accessible mode
explicitly, rather than relying on the `TERM=dumb` auto-detection)
gives a clean, reliable pipe-testable path for `RunS3Menu` and the
three bucket-selection call sites above. Resolve this first -- if it
works, use it for all four huh call sites in this piece of work AND
retroactively for `object_browser.go`; if it turns out genuinely not
pipe-testable, say so explicitly (matching how PLAN.md documented the
bubbletea-Model testing question before teatest resolved it) rather
than silently shipping more untested huh code.
