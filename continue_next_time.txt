NEXT UP: full-height domain picker (the deferred half of today's first request)
--------------------------------------------------------------------

The user's actual request was "clear the screen first and take up the
full height of the terminal window." Only the clear-screen half was
implemented (Phase 20.24) -- the domain picker itself is still a
compact, content-sized huh.Select, not full-height. This was
deliberately NOT guessed at further, because it's a real architectural
fork, not an implementation detail:

- The Menu tier (huh.Select) has no built-in "occupy the full terminal
  height" concept the way the Picker/List/Manager tier's bubbletea
  components do (real tea.WindowSizeMsg-driven sizing, content padded
  to fill).
- Making ONLY the root domain picker full-height would look
  inconsistent with every menu one level deeper (S3/EC2/Key Management
  submenus), which would stay compact -- the opposite direction from
  all the chrome-consistency work just finished.
- The much bigger alternative -- redesigning the entire Menu tier onto
  a full-height bubbletea component, effectively retiring huh.Select
  for navigation menus -- hasn't been scoped at all.

Ask the user directly which of these (or something else entirely) they
meant by "full height" before writing any code. See DECISIONS.md,
"Clear the screen at startup," and PLAN.md Phase 20.24's own deferred
note for what's already been reasoned through.
