Manual test plan: awsops against real AWS
=============================================

Purpose: verify the Go awsops binary (see cmd/awsops, internal/workflow)
against real AWS before retiring ec2_ami_manager.bash, ami_copy.bash, and
tests/*.bats (see DECISIONS.md, "Retarget implementation from Bash to
Go" and PLAN.md Phase 17's retire-after-verify plan). Everything so far
has only been verified against fakes in unit tests -- no run in this
plan has touched real AWS yet.

Prerequisites
--------------
- `make build` (or `go build -o bin/awsops ./cmd/awsops`) produces a
  binary that runs `-help`/`-version`/`-license` correctly.
- AWS credentials configured and resolvable by the SDK's default chain
  (env vars, ~/.aws/credentials, or SSO) with the IAM permissions listed
  in DESIGN.md, "Assumptions" #2 (Describe*/Run/Start/Stop/Terminate/
  CreateImage/DeregisterImage/CreateTags/DeleteTags/DescribeTags/
  DescribeInstanceAttribute/DescribeVolumes on the EC2 side,
  SendCommand/GetCommandInvocation/DescribeInstanceInformation on the
  SSM side, HeadObject on the S3 side).
- At least one disposable/low-cost test instance (ideally in a test
  account or clearly-marked test resource, not production) in a region
  awsops is configured for (internal/awsclient/regions.go: us-east-1,
  us-east-2, us-west-1, us-west-2). A plain small instance is fine for
  exercising the mechanics; a real Invenio RDM instance is the realistic
  target for Feature 7 (Create AMI) and Feature 10 (Backup Archive).
- SSM Agent running on at least one test instance, so the fstrim/
  cloud-init-check/backup paths get exercised (not just their
  "SSM unavailable" fallbacks).
- A second test instance with SSM either absent or stopped, to exercise
  the "SSM unavailable" fallback paths deliberately.
- For Feature 10 (Backup Archive & Trim) specifically: an S3 bucket for
  archival, and the target instance's IAM instance profile granting
  s3:PutObject (and likely s3:ListBucket scoped to its prefix) on that
  bucket. As of this writing neither exists yet (see DESIGN.md,
  Assumptions #3-4) -- skip section 10 below until both are in place,
  and don't guess at a bucket name or work around the gap.

General checks (do these once, up front)
------------------------------------------
- Startup: awsops prints "awsops <version> -- authenticated as AWS
  account <account-id>" before the first listing. If credentials aren't
  resolvable, it should fail fast with a clear message (DESIGN.md,
  Error Handling Strategy #4) -- not a raw SDK stack trace.
- `-debug`: run `awsops -debug`, confirm it prints "Debug log:
  ./awsops-debug-<timestamp>.jsonl" to stderr before the first listing,
  and that file exists and grows as you use the tool. `tail -f` it
  while performing a few operations (e.g. list refresh, start/stop an
  instance): confirm each line is valid JSON (`jq . <file>` on the
  whole file at the end is a quick way to check), each has "time",
  "event" ("aws_call"), "method" (e.g. "EC2.DescribeInstances",
  "SSM.SendCommand"), "region", and either "output" or "error". Run
  without `-debug` (the default): confirm no debug file is created and
  no debug-related output appears anywhere.
- The initial listing shows "===== CURRENT EC2 INSTANCES =====" and
  "===== AVAILABLE AMIs (owned by account) =====" tables, aggregated
  across all four regions in one view (not one region at a time --
  DECISIONS.md, "All four regions aggregated in unified view").
  Untagged Project/Environment show as "unknown"; untagged Name shows
  blank. If your terminal is a color-capable TTY and NO_COLOR isn't set,
  the STATE column should be colorized (running=green, stopped/
  terminated=red, pending/stopping=yellow); redirecting output to a file
  or setting NO_COLOR=1 should produce plain text with no escape codes.
- The main menu shows exactly these 12 options, in this order:
    1) [ok] Create EC2 instance from AMI
    2) [ ] Create EC2 instance from cloud-init YAML
    3) [ ] Start EC2 instance
    4) [ ] Stop EC2 instance
    5) [ ] Terminate EC2 instance
    6) [ ] Manage tags for an instance or AMI
    7) [ ] Create AMI from EC2 instance (running or stopped)
    8) [ok] Remove AMI
    9) [ ] Show/export cloud-init for an instance or AMI
   10) [ ] Archive stale backups to S3 and trim disk space
   11) [ok] Refresh resource lists
   12) [ok] Exit
- Choosing "Refresh resource lists" re-fetches and re-displays both
  tables. Every other successful operation (1-10) should also trigger
  an automatic refresh afterward (DECISIONS.md, "Refresh data after
  each operation").
- A single operation's error (e.g. cancel a pick list mid-flow by typing
  an out-of-range number twice, or let an AWS call fail) should print
  "Error: ..." and return to the main menu -- not crash the whole
  process. AWS API errors should show as "AWS error [Code]: Message"
  (Phase 15's formatError), not a raw wrapped Go error string.
- Ctrl+C while a menu/sub-prompt is active should print "Exiting." and
  exit 0 cleanly. Ctrl+C while a long operation is polling (e.g. mid-AMI
  creation) should also exit cleanly rather than hang or panic.
- If your account or any region has more than 50 instances or AMIs, the
  pick list should paginate ('n'/'p' to move between pages) rather than
  dumping everything at once; item numbers must stay valid across pages
  (Phase 15, pagination).

1. Create EC2 instance from AMI (menu option 1)
--------------------------------------------------
Prompts, in order: pick an AMI -> "Name tag" -> "Instance type" [t3.micro]
-> "Key pair name" -> a numbered security group list plus "Security
group IDs (numbers above and/or raw IDs, comma-separated)" -> "Select a
subnet" -> "IAM instance profile (optional)" -> "User data (inline text,
@file path, or blank)" -> "Project tag" [defaults to the AMI's Project
tag if it has one] -> "Environment tag (production, development, or
test)" -> confirm "Launch this instance?".

- Name tag is prompted right after picking the AMI, before any
  technical configuration -- moved up from the end of the flow per
  feedback that naming the instance should come early.
- Security groups and subnet are pick lists fetched from the AMI's
  region (ec2:DescribeSecurityGroups/DescribeSubnets) -- this replaces
  the earlier free-text-only UX where typing a security group *name*
  instead of an ID caused a real `InvalidParameterCombination` AWS
  error. Key pair name stays free text: unlike opaque `sg-.../subnet-...`
  IDs, key pair names are already human-readable, and a flat list of
  every key pair in the account (regardless of relevance to this AMI)
  turned out to be noise rather than help.
- Security groups accept a comma-separated mix of list numbers and raw
  `sg-...` IDs (e.g. "1,2" or "1,sg-0abc123"). Type an out-of-range
  number: confirm it's rejected ("invalid selection ...") and re-prompts.
  Leave it blank: confirm "at least one security group ID is required"
  and re-prompts.
- If your account/region has zero subnets or zero security groups
  (unlikely, but worth spot-checking once), confirm each falls back
  cleanly to its old free-text prompt instead of showing an empty,
  unusable list.
- Leave Name tag or Key pair name blank once each: confirm each is
  rejected ("invalid input: this field is required and cannot be
  blank") and re-prompts -- note the message is field-agnostic, not the
  old hardcoded cloud-init-YAML wording that used to leak into
  unrelated prompts.
- Type an Environment value other than production/development/test:
  confirm it's rejected and re-prompts.
- Point User data at a real file from a local clone of
  caltechlibrary/cloud-init-examples via "@/path/to/file.yaml": confirm
  the file's contents (not the literal string) get used.
- Decline the final confirmation once: confirm "Cancelled." and no
  instance is created.
- Accept: confirm the instance launches, awsops polls until running
  (bounded, ~5 minutes) and prints connection info (public/private IP,
  ssh command if a public IP exists).
- If you supplied user data: confirm awsops waits for SSM Online, runs
  `cloud-init status --wait`, and reports "cloud-init completed
  successfully" or an error status -- not just that the instance reached
  running. If SSM never comes online, confirm it says so and skips
  cleanly (not an error).
- Confirm the new instance's tags in the AWS console match what you
  entered (Name/Project/Environment), and that its region matches the
  AMI's region.

2. Create EC2 instance from cloud-init YAML (menu option 2)
----------------------------------------------------------------
Prompts, in order: "Cloud-init YAML (inline text or @file path)"
(required, non-empty) -> pick a base AMI -> the same
Name/InstanceType/KeyPair/SecurityGroups-pick-list/Subnet-pick-list/
IAMProfile/Project/Environment prompts as option 1, in the same order --
but no separate user-data prompt, since the cloud-init YAML collected
first *is* the user data.

- Leave the cloud-init prompt blank: confirm it's rejected and
  re-prompts (this is the one workflow where user data is mandatory,
  not optional).
- Point it at a real file via "@/path/...": confirm the base AMI pick
  list appears *after* the cloud-init prompt, not before.
- Accept through to launch: confirm it shares the exact same execution
  path as option 1 (poll until running, then SSM + cloud-init-status
  check) -- this should behave identically to option 1's user-data path,
  just with a different prompt order.

3. Start EC2 instance (menu option 3)
------------------------------------------
- With no stopped instances anywhere: confirm "No stopped instances
  found." and a clean return to the menu (no pick list shown).
- With at least one stopped instance: pick list shows only stopped
  instances. Confirm "Start instance <id> (<name>)?", decline once
  (confirm nothing happens), then accept: confirm ec2:StartInstances is
  called, awsops polls until running, and displays connection info plus
  the note that the public IP may have changed since it uses an Elastic
  IP unless one is attached.

4. Stop EC2 instance (menu option 4)
-----------------------------------------
- With no running instances: confirm "No running instances found."
- With at least one running instance: pick list shows only running
  instances. Confirm "Stop instance <id> (<name>)?", decline once, then
  accept: confirm ec2:StopInstances is called, awsops polls until
  stopped, and prints "Instance <id> is now stopped." (no connection
  info, since a stopped instance has none).

5. Terminate EC2 instance (menu option 5) -- destructive, be careful
-------------------------------------------------------------------------
Use a genuinely disposable test instance for this section.

- Pick any instance (running or stopped -- both are valid targets).
  Confirm the dry-run header "=== DRY RUN: terminating <id> (<name>)
  ===" appears before anything is touched.
- If the instance has any EBS volume with DeleteOnTermination=true,
  confirm the WARNING naming the affected device(s) appears; if none do,
  confirm no such warning appears.
- If the instance is tagged Environment=production, confirm the extra
  "WARNING: this instance is tagged Environment=production." appears
  before type-to-confirm; confirm it's absent otherwise.
- At the type-to-confirm prompt, type something wrong once: confirm it
  cancels immediately with no retry loop (this is deliberate -- a
  mismatch cancels rather than re-prompting, unlike every other
  validation in this tool). Re-run and type the exact instance ID: it
  should terminate. Re-run once more and type the instance's Name tag
  instead of its ID: confirm that's also accepted (DESIGN.md says
  "ID or name").
- Confirm the instance actually terminates in the AWS console.

6. Manage tags for an instance or AMI (menu option 6)
----------------------------------------------------------
Prompts: pick "Instance" or "AMI" -> pick the specific resource ->
displays its current tags -> pick "Add"/"Update"/"Remove" -> (Add: "New
tag key" [required, non-empty] then "New tag value"; Update: pick an
existing key from a list, then a new value defaulted to the current
one; Remove: pick an existing key from a list) -> confirm.

- Run this once against an instance and once against an AMI to cover
  both resource types (DESIGN.md explicitly calls out that AMI tag
  editing was a real gap before this tool).
- Add: leave the key blank once, confirm it's rejected and re-prompts;
  then supply a real key/value, confirm, and verify via
  `aws ec2 describe-tags` (or the console) that it landed.
- Update: pick an existing tag, confirm the new-value prompt shows the
  current value as the default (pressing Enter keeps it unchanged).
- Update the Environment tag specifically: confirm the on-screen note
  ("Environment is the same tag this tool uses elsewhere... to gate
  production warnings") appears.
- Remove: pick an existing tag, confirm, verify it's actually gone.
- With a resource that has no tags at all, try Update or Remove: confirm
  "No existing tags to update/remove." and a clean return, no crash.
- Decline the final confirmation once: confirm no CreateTags/DeleteTags
  call happens (check the console/CLI to be sure nothing changed).
- Confirm this never touches an AMI's Name *attribute* -- only tags. An
  AMI's Name in the console should be unchanged even after editing its
  tags.

7. Create AMI from EC2 instance (menu option 7)
----------------------------------------------------
This is the most involved workflow -- budget real time for AMI creation
itself (see the estimate table below).

- Pick a stopped instance first. Confirm the volume-size summary lists
  total attached-volume size and an estimated creation time bucket
  matching: <20GB -> 5-15 min, 20-100GB -> 15-45 min, 100-200GB -> 45-90
  min, 200GB+ -> 1.5-3+ hours. If any volume has a prior snapshot,
  confirm the "only changed blocks will be copied" note appears.
- Confirm NO crash-consistency guidance and NO fstrim offer appear for a
  stopped instance (those are running-instance-only).
- Repeat against a running instance: confirm the crash-consistency text
  (PostgreSQL/OpenSearch WAL replay, Redis ephemeral loss, Docker
  images unaffected) appears before the fstrim offer.
  - If SSM is online: confirm "Run fstrim via SSM before snapshotting?"
    appears; accept once and confirm it reports either the trim output
    or a "did not complete" status, never silently doing nothing.
  - Against your second (SSM-absent) instance: confirm "SSM is not
    available on this instance; skipping fstrim." and no prompt at all
    for the fstrim step -- a clean skip, not an error.
- AMI name prompt: confirm the default suggestion is
  "<name-or-id>-copy-<today's date>". Try clearing it and typing an
  invalid name (e.g. containing a semicolon, or 2 characters): confirm
  it's rejected with a message about the 3-128 character / allowed-
  character constraint, then accept a valid name.
- No-reboot: only offered for a running instance; confirm it's skipped
  entirely for a stopped one.
- Project/Environment: confirm Project defaults to the source instance's
  Project tag if it has one.
- Accept the final confirmation: confirm AMI creation starts, and that
  awsops prints a periodic "... waiting for AMI to become available
  (elapsed ...)" status line (Phase 15's loading indicator) rather than
  going silent for the whole duration.
- This is the single most important regression check for this whole
  project: confirm the poll does NOT give up after 10 minutes. Watch it
  through to completion (or budget the real time from the estimate
  above) and confirm it reports "AMI <id> is now available (took
  <duration>)." with a correct elapsed-time format.
- Confirm the resulting AMI's ID, name, state, creation date, and region
  match the AWS console, and that its Tags include the Name/Project/
  Environment you entered.
- If feasible, deliberately trigger a failure (e.g. terminate the source
  instance mid-creation, or use insufficient permissions) and confirm
  awsops reports "AMI <id> creation failed after <duration>." rather
  than hanging or giving a generic timeout.

8. Remove AMI (menu option 8) -- destructive, be careful
--------------------------------------------------------------
Use an AMI you created in section 7 for this, not a production AMI.

- Pick an AMI. Confirm the dry-run header "=== DRY RUN: removing AMI
  <id> (<name>) ===" appears.
- If any instance still references this AMI's ImageId, confirm a
  WARNING lists it by instance ID and name; pick an AMI with no
  dependents and confirm no such warning appears.
- If the AMI is tagged Environment=production, confirm the extra
  production warning appears before type-to-confirm; confirm it's absent
  otherwise.
- Type-to-confirm: try a wrong value once (confirm it cancels
  immediately, no retry), then the exact AMI ID (confirm it works), then
  repeat once more typing the AMI's Name instead (confirm that's also
  accepted).
- Confirm the AMI is actually deregistered in the console.

9. Show/export cloud-init for an instance or AMI (menu option 9)
------------------------------------------------------------------------
Prompts: pick "Instance" or "AMI".

Instance path (free, instant):
- Pick an instance that was launched with user data (e.g. one from
  section 1 or 2): confirm the exact decoded cloud-init/user-data text
  is displayed between "--- cloud-init ---" markers.
- Pick an instance launched with no user data: confirm "No user-data was
  set at launch for <id>." and a clean return -- not an error, and no
  export prompt.
- After a successful display, confirm the "Save to file (blank to
  skip)" export prompt appears; leave it blank once (confirm nothing is
  written), then re-run and give it a real path (confirm the file is
  written with the exact decoded content, and "Saved to <path>"
  appears). Diff the exported file against your local clone of
  caltechlibrary/cloud-init-examples to check for drift -- this is the
  actual point of this feature.

AMI path (costs real time and a temporary instance -- confirm carefully):
- Pick an AMI. Confirm the explicit warning ("Extracting cloud-init from
  an AMI launches a temporary billable instance. Proceed?") appears and
  requires an explicit accept -- decline once and confirm no instance is
  launched.
- Accept: confirm awsops launches a small (t3.micro) disposable
  instance tagged Purpose=cloud-init-extraction, shows the "...
  extracting cloud-init..." periodic status line, waits for running and
  SSM Online (bounded timeout -- this should fail cleanly within a few
  minutes if SSM never comes online, not hang indefinitely), reads
  /var/lib/cloud/instance/user-data.txt via SSM, and displays it.
- Critical check: after the extraction finishes (success, SSM timeout,
  or command failure -- try to hit at least the success case and one
  failure case), confirm in the AWS console that the temporary instance
  was actually terminated. This "always terminate" guarantee is the
  whole point of this workflow's design; if a temporary instance is
  left running after a failure, that's a real bug, not a cosmetic one.

10. Archive stale backups to S3 and trim disk space (menu option 10)
--------------------------------------------------------------------------
BLOCKED as of this writing: the S3 bucket and the target instance's IAM
instance profile grant (s3:PutObject/s3:ListBucket) don't exist yet (see
DESIGN.md Assumptions #3-4). Do not attempt this section until both are
in place. Once they are, test against a throwaway instance launched from
an existing AMI that already has real backup files baked in -- never
directly against a production instance (per PLAN.md Phase 16's own
constraint).

Prompts, in order: pick an instance -> "Backup directory (e.g.
/opt/rdm_sql_backups)" -> "Age threshold in days" -> "S3 bucket" ->
dry-run list -> type-to-confirm (instance ID or name).

- Leave directory/bucket blank once each: confirm rejected and
  re-prompted. Type a non-numeric or zero/negative age threshold:
  confirm rejected and re-prompted.
- Confirm the dry-run lists exactly the files older than the threshold,
  with size and age, before anything is touched -- and that it excludes
  files younger than the threshold.
- With zero files matching the threshold: confirm "No files match the
  age threshold. Nothing to do." and a clean return with no
  type-to-confirm step at all.
- Type-to-confirm: wrong value cancels immediately (no retry); correct
  instance ID or name proceeds.
- Confirm the upload phase shows the "... uploading backup files to S3"
  periodic status, then the "... verifying uploads via s3:HeadObject"
  status -- these are two distinct phases, not one.
- Independent verification is the safety-critical part of this feature:
  if you can arrange a partial failure (e.g. revoke the instance's S3
  permission mid-upload, or use a bucket the instance can't reach),
  confirm that only the files the tool's own HeadObject call actually
  verified get deleted -- unverified files must be left on disk and
  named in the final report, never silently dropped or deleted based on
  the instance's own self-report.
- Confirm fstrim runs after deletion regardless of whether every file
  verified (partial success should still trigger fstrim on whatever was
  actually freed).
- Confirm the final report states the count of files freed, total bytes
  freed, and lists any files that failed verification.
- Verify in the S3 console that the uploaded objects exist with the
  correct size, and on the instance that only the verified files were
  actually removed from disk.

Pass/fail
----------
If every section above behaves as described, the Go binary has reached
parity with (and fixed the known bugs in) ec2_ami_manager.bash/
ami_copy.bash across all four regions, and PLAN.md Phase 17 (retire
ec2_ami_manager.bash, ami_copy.bash, ami_copy_basic_steps.md, and
tests/*.bats) can proceed.

If anything regresses (a prompt that doesn't validate what it should, a
poll that gives up too early, a temporary resource left behind, an
error message that isn't actionable), note it in PLAN.md Phase 16 or
TODO.md and fix it before retiring the Bash scripts -- don't delete them
until the gap is closed. This is the same retire-after-verify discipline
this project has followed since ami_copy.bash (DECISIONS.md, 2026-06-30).
