Manual test plan: clasm against real AWS
=============================================

Purpose: verify the Go clasm binary (see cmd/clasm, internal/workflow)
against real AWS before retiring ec2_ami_manager.bash, ami_copy.bash, and
tests/*.bats (see DECISIONS.md, "Retarget implementation from Bash to
Go" and PLAN.md Phase 17's retire-after-verify plan). Everything so far
has only been verified against fakes in unit tests -- no run in this
plan has touched real AWS yet.

Prerequisites
--------------
- [x] `make build` (or `go build -o bin/clasm ./cmd/clasm`) produces a
  binary that runs `-help`/`-version`/`-license` correctly.
- [x] AWS credentials configured and resolvable by the SDK's default chain
  (env vars, ~/.aws/credentials, or SSO) with the IAM permissions listed
  in DESIGN.md, "Assumptions" #2 (Describe*/Run/Start/Stop/Terminate/
  CreateImage/DeregisterImage/CreateTags/DeleteTags/DescribeTags/
  DescribeInstanceAttribute/DescribeVolumes on the EC2 side,
  SendCommand/GetCommandInvocation/DescribeInstanceInformation on the
  SSM side, HeadObject on the S3 side).
- [x] At least one disposable/low-cost test instance (ideally in a test
  account or clearly-marked test resource, not production) in a region
  clasm is configured for (internal/config's DefaultRegions, used when
  no ~/.clasm file exists: us-west-1, us-west-2). A plain small
  instance is fine for
  exercising the mechanics; a real Invenio RDM instance is the realistic
  target for Feature 7 (Create AMI) and Feature 10 (Backup Archive).
- [x] SSM Agent running on at least one test instance, so the fstrim/
  cloud-init-check/backup paths get exercised (not just their
  "SSM unavailable" fallbacks).
- [x] A second test instance with SSM either absent or stopped, to exercise
  the "SSM unavailable" fallback paths deliberately.
- [x] For Feature 10 (Backup Archive & Trim) specifically: an S3 bucket for
  archival, and the target instance's IAM instance profile granting
  s3:PutObject (and likely s3:ListBucket scoped to its prefix) on that
  bucket. As of this writing neither exists yet (see DESIGN.md,
  Assumptions #3-4) -- skip section 10 below until both are in place,
  and don't guess at a bucket name or work around the gap.

2. Create EC2 instance from cloud-init YAML (menu option 2)
----------------------------------------------------------------
Prompts, in order: "Cloud-init YAML file path" (required, non-empty) ->
pick a base AMI -> the same
Name/InstanceType/KeyPair/SecurityGroups-pick-list/Subnet-pick-list/
IAMProfile/Project/Environment prompts as option 1, in the same order --
but no separate user-data prompt, since the cloud-init YAML collected
first *is* the user data. Unlike option 1's optional "User data" field,
this prompt no longer accepts inline text at all -- it always reads from
a file (DECISIONS.md, "Create EC2 Instance from Cloud-Init YAML always
reads from a file"), a follow-up to this session's real bug where typing
a bare filename (e.g. `newt-machine.yaml`) without the `@` prefix
silently became the literal user-data string instead of the file's
contents.

- [x] Leave the cloud-init prompt blank: confirm it's rejected and
  re-prompts (this is the one workflow where user data is mandatory,
  not optional).
- [x] Type a real file's path directly, with no "@" prefix (e.g.
  `newt-machine.yaml` for a file in the current directory): confirm it
  loads that file's contents and the base AMI pick list appears
  *after* the cloud-init prompt, not before. This should just work now
  without needing "@" at all.
- [x] Type the same real file's path *with* a leading "@" (old muscle
  memory from option 1's User data prompt): confirm it's tolerated --
  the "@" is stripped and the file loads normally.
- [x] Type a path to a file that doesn't exist (e.g.
  `/no/such/file.yaml`): confirm it's rejected with `invalid input:
  cannot read "...": ...` and re-prompts, rather than silently using the
  path string as literal user-data (the old bug's failure mode).
- [x] Accept through to launch: confirm it shares the exact same execution
  path as option 1 (poll until running, then SSM + cloud-init-status
  check) -- this should behave identically to option 1's user-data path,
  just with a different prompt order.
